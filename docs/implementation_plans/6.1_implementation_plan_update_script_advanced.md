# Walkthrough: Update Script v2

We have fully implemented the feedback from the other LLM and upgraded the toolkit to **v2** of the deployment and update architecture!

## What We Built

### 1. The Great Template Path Standardization
The other LLM correctly pointed out a critical inconsistency in the toolkit: our prompts (`PROMPT_USAGE.md`) expected templates to live in `.forge/templates/`, but `install.ps1` and `README.md` were telling users to put them in `templates/` at the root of their repo. 
We fixed this by establishing `.forge/templates/` as the single source of truth across the entire ecosystem:
- Updated **`install.ps1`** to scaffold templates into `.forge/templates/`.
- Updated **`README.md`** instructions.

### 2. Automatic Legacy Migration
Because users running older versions of Copilot Forge will have a legacy `templates/` folder at their repo root, `update.ps1` now performs an automatic migration.
During an update, if it detects `templates/`, it will safely move it into `.forge/templates/` before it begins the sync, keeping the user's project clean and fully compliant with the new standard.

### 3. `-HealthOnly` Audit Mode
We added a new switch to `update.ps1`: `-HealthOnly`. 
When run, the script acts strictly as a read-only preflight audit tool. It performs zero copying and zero version stamping. It simply reads the source and target files, computes the MD5 hashes, generates the template drift diffs, and outputs a clean Health Report showing the version gap, any protected-file conflicts, and template drift.

### 4. `-JsonReport` CI Integration
For teams running Copilot Forge in enterprise environments, we added a `-JsonReport` flag. This suppresses all the human-readable console colors and tables, and instead dumps a deeply nested, structured JSON object containing the full update summary (Files Added, Skipped, Template Drift stats, etc.), making it trivial to pipe the output into a CI pipeline for automated compliance checking.

### 5. Config Awareness
The update script now explicitly checks for the existence of `.forge/config.yml` in the target directory to ensure it is actually updating a valid Forge project before touching any files.

---

> [!TIP]
> The update mechanism is now a fully matured, production-grade deployment CLI. You can use `.\update.ps1 -TargetDir "C:\my-repo" -HealthOnly -JsonReport` in your CI to automatically detect if developers are drifting from your canonical Forge standards!
