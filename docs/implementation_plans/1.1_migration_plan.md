# Migration Path for Feature 14 (Corpus/Guide Split)

You raise an excellent point! If an existing user copies the new `.github/prompts/` to their project, their local `.forge/context/` will still have the old monolithic structure. The new prompts will look for `rules/` and `corpus/` and fail. 

To solve this, we need to build a migration path. Furthermore, I realized that I missed updating the `#init` prompt to generate the new structure for *new* projects! We will fix both issues at once.

## Proposed Changes

### Update `init.prompt.md`
We will modify the `#init` prompt (which is responsible for bootstrapping the `.forge/` directory) to handle both new initializations and legacy migrations.

1. **Update Directory Scaffold for New Users**:
   Update Step 3 in `init.prompt.md` to create the new structure:
   ```text
   .forge/
     context/
       project-overview.md
       taxonomy.md
       rules/
         architecture.rules.md
         conventions.rules.md
         ...
       corpus/
         architecture.md
         conventions.md
         ...
   ```

2. **Add an Auto-Migration Step for Existing Users**:
   We will add a new step to the `#init` instructions specifically for handling legacy repositories.
   If `#init` detects the presence of `.forge/context/architecture.md` (the legacy monolithic file), it will:
   - Use the terminal to create the `rules/` and `corpus/` directories.
   - Move the legacy files (`architecture.md`, `conventions.md`, `variables.md`, `deployment.md`) into `corpus/`.
   - Read the newly moved files and extract the core directives to generate `rules/architecture.rules.md` and `rules/conventions.rules.md`.
   
   This way, all an existing user needs to do after updating their toolkit is run `#init`, and Copilot will automatically transform their old knowledge into the new tiered structure!
