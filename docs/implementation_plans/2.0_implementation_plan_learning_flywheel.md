# Feature 15: Learning Flywheel (Self-Improving Knowledge)

The goal of this feature is to turn the current passive knowledge capture system into an active self-improving "flywheel". We will implement an additive cycle (auto-capturing surprises to an inbox) and a subtractive cycle (pruning stale rules).

## Proposed Changes

### 1. Directory Scaffolding Update (`init.prompt.md`)
We will update `init.prompt.md` to automatically generate two new directories inside `.forge/knowledge/`:
- `inbox/`: For unprocessed learning candidates.
- `archive/`: For pruned or rejected knowledge.

### 2. Auto-Capture Logic in Core Prompts
We will update the core pipeline prompts to actively identify new knowledge gaps and write them to `.forge/knowledge/inbox/`:
- **`reflect.prompt.md`**: Instruct it to write a candidate if it encounters a convention violation that is *not* covered by the existing `.rules.md` files.
- **`review.prompt.md`**: Instruct it to write a candidate if it notices an anti-pattern or missing architectural constraint.
- **`wrapup.prompt.md`**: Enhance its current knowledge capture to write explicitly shaped learning candidates to the `inbox/` instead of directly injecting them into lessons.

### 3. New Prompt: `#synthesize`
Create `.github/prompts/synthesize.prompt.md`. This agent will act as the inbox processor. For each file in `inbox/`, it will:
1. Classify the candidate (lesson, ADR, convention update, reject).
2. If it's a **lesson**, use `lesson-template.md` to save it to `lessons/`.
3. If it's a **convention update**, propose a diff to the relevant `.rules.md` file.
4. If it's a **rejection**, move it to `archive/`.

### 4. New Prompt: `#prune`
Create `.github/prompts/prune.prompt.md`. This agent acts as the garbage collector for stale knowledge. It will:
1. Read `.forge/knowledge/lessons/` and `.forge/context/rules/`.
2. Compare them against the current codebase empirically.
3. Move stale or superseded items to `archive/` with reasoning.
4. Produce a hygiene report for the developer.

## Open Questions

> [!IMPORTANT]
> The roadmap states that the `#synthesize` prompt should propose diffs to the `.rules.md` files. Should the `#synthesize` prompt automatically apply these changes to the rules files, or should it just present the proposed changes for the user to manually accept/reject in the chat? I recommend presenting them for user review first to maintain control over the rules.
